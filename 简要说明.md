# 产地加工场景 Demo 简要说明

## 一、业务场景理解、库存和成本处理方案

我将这个场景理解为水果加工厂中的一个“加工批次结转”问题。

一批原料果进入加工环节后，不会只产生一种结果，而是会被分选、打包成多种规格的成品果，例如 5 斤装、8 斤装、10 斤装；同时还会产生次果和损耗。因此，这个场景的重点不是单纯录入数据，而是要同时处理两类变化：

```text
1. 库存变化：原料减少，成品和次果增加，损耗被记录。
2. 成本变化：原料成本需要结转到最终加工结果中。
```

### 库存处理逻辑

我把一次加工看作一个独立批次，库存按以下方式处理：

| 类型 | 库存处理 | 说明 |
| --- | --- | --- |
| 原料果 A | 库存减少 | 加工投入，被消耗 |
| 原料果 B | 库存减少 | 加工投入，被消耗 |
| 成品果 | 库存增加 | 按 5 斤装、8 斤装、10 斤装箱数折算入库重量 |
| 次果 | 库存增加 | 作为低等级产出入库 |
| 损耗 | 不入库 | 只记录重量、损耗率和损耗成本 |

系统会校验投入和产出是否合理：

```text
总投入重量 = 原料果 A 重量 + 原料果 B 重量
总产出重量 = 成品果重量 + 次果重量 + 损耗重量
```

如果总产出重量大于总投入重量，说明录入数据不合理，demo 中会禁止查看结果；如果总产出重量小于总投入重量，说明存在未分配重量，页面会提示复核，并支持将差额计入损耗，使批次重量闭环。

### 成本处理逻辑

题目给定：

```text
原料果 A 成本价：2 元/斤
原料果 B 成本价：3 元/斤
```

demo 中只处理原料成本，暂不考虑人工、包材、水电、设备折旧等加工费用。

成本计算逻辑如下：

```text
原料 A 成本 = 原料 A 重量 × 2
原料 B 成本 = 原料 B 重量 × 3
原料总成本 = 原料 A 成本 + 原料 B 成本
加权单位成本 = 原料总成本 ÷ 总投入重量
```

由于原料 A、B 在加工后已经混合，demo 中不追踪具体某一斤原料进入了哪个规格，而是采用加权平均成本进行结转。加工结果按重量分摊原料成本：

```text
成品果成本 = 成品果重量 × 加权单位成本
次果成本 = 次果重量 × 加权单位成本
损耗成本 = 损耗重量 × 加权单位成本
```

其中，成品果和次果进入库存，因此它们的分摊成本作为入库成本；损耗不进入库存，但它消耗了原料，所以单独展示为加工损耗成本。

这样处理后，可以形成两个闭环：

```text
投入重量 = 成品果重量 + 次果重量 + 损耗重量
原料总成本 = 成品果成本 + 次果成本 + 损耗成本
```

## 二、主动补充的规则和假设

题目没有给出完整商业系统规则，因此我在 demo 中主动补充了以下规则和假设：

1. 一次录入代表一个完整加工批次。
2. 原料果 A、B 混合加工后，不区分具体流向到哪个成品规格。
3. 成本只考虑题目给定的原料成本，不考虑人工、包材、水电等额外费用。
4. 成品规格固定为 5 斤装、8 斤装、10 斤装，不做动态规格维护。
5. 次果虽然不是标准成品，但仍有可销售、可二次加工或可处理价值，因此作为低等级库存产出。
6. 损耗没有库存价值，不入库，但需要记录重量、损耗率和损耗成本。
7. 当产出重量小于投入重量时，差额作为未分配重量提示用户复核，并支持一键计入损耗。
8. 当产出重量大于投入重量时，认为录入数据不合理，禁止进入结果页。
9. demo 使用前端模拟数据，不接入数据库。
10. 页面内置默认示例数据，方便打开后直接演示完整流程。

| ChatGPT | 用于进一步梳理业务规则、库存处理逻辑、成本结转方案，以及整理说明文档中的业务表达。 |
| Codex | 用于项目落地实现，包括 React + TypeScript 页面开发、移动端样式调整、README 编写、GitHub Pages 部署配置、本地构建检查和线上部署验证。 |

AI 工具主要起到辅助分析、辅助规划、辅助编码和辅助检查的作用。最终采用的业务规则，例如次果入库、损耗不入库但承担成本、原料成本按加权平均方式结转等，是基于题目场景和业务合理性做出的主动判断。
